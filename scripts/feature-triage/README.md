# Feature Request Triage

A small, re-runnable system that pulls open feature requests, triages them
(category · scope · effort · priority · assessment), and renders a living doc
at [`docs/feature-triage.md`](../../docs/feature-triage.md).

It doubles as a **release-notes engine**: the "Recently Shipped" section lists
completed requests with credit to the original requester.

## Pieces

| File | Role |
|------|------|
| `fetch.py` | Pulls active (open/planned/beta) + recently-shipped requests **with comments** from the API into `data/feature-requests.json`. |
| `triage.json` | Curated, version-controlled annotations keyed by request id (category, scope, effort, confidence, rationale, risks, possible-dup). **The judgment layer.** |
| `generate.py` | Merges live data + `triage.json` → `docs/feature-triage.md`. Completion auto-syncs from live DB status. |
| `data/` | Raw API snapshot cache (git-ignored — regenerated by `fetch.py`). |

## Refresh the doc

```bash
cd scripts/feature-triage
python3 fetch.py        # pull latest requests + comments
python3 generate.py     # rebuild docs/feature-triage.md
```

If new requests appeared since the last run, `generate.py` lists them under
**⚠️ Untriaged**. Add an entry for each to `triage.json` (see the shape of the
existing entries), then re-run `generate.py`.

Only Python 3 stdlib is used — no dependencies.

## How things are decided

- **Category** — `quick_win` (low effort, clear value) · `easy` (cheap, modest
  demand) · `full_feature` (substantial build) · `probably_not` (out of scope /
  infeasible / cost ≫ value). This is human/LLM judgment, grounded in the code.
- **Scope** — which of the 4 surfaces a correct implementation touches:
  🌐 website · ⚙️ API · 📱 mobile · 🤖 bot.
- **Priority** — `upvotes + 2×comments` (comments weighted: they signal engaged,
  detailed demand). Sorted within each category.
- **Completion** — derived from live DB status, never stored in markdown:
  `released` → shipped/checked, `beta_testing` → in beta, `planned` → planned.
  Re-running never loses "done" state.

## Data access

`fetch.py` calls the same public read endpoints the website uses
(`GET /api/v2/feature-requests`, `GET /api/v1/feature-requests/{id}`). First-party
access is granted by the `Origin` header (see `api/handlers/gateway.go`) — this is
not a secret; it's how our own web clients reach their own API. Override host /
origin with env vars if needed:

```bash
FR_API_BASE=https://police-cad-api-dev.herokuapp.com \
FR_API_ORIGIN=https://police-cad-dev.herokuapp.com \
python3 fetch.py
```

Other env knobs: `FR_STATUSES` (default `open,planned,beta_testing`),
`FR_SHIPPED_STATUSES` (default `released`), `FR_SHIPPED_LIMIT` (default 60),
`FR_SITE_BASE` (link base for `generate.py`).
